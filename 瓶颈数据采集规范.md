## 1. 总体流程

```text
Isaac Sim 仿真（rule_based 主循环，逻辑不变）
    ↓
HcSingleEnvBase.step_env_logic 末尾
    ↓ BottleneckDataCollector（新增，只读 env_state_action_dict）
原始日志 7 表（本目录 output/bottleneck_dataset/<run_id>/）
    ↓ 离线脚本（后续）
window_feature_table.csv  →  bottleneck_label.csv
    ↓
GNN / 时序模型输入
```

采集原则（与调研文档一致）：

1. **事件驱动优先**：`resource_event_log`、`buffer_event_log` 在状态变化时写入；`job_trace`、`route_transport_task` 在任务边界写入。
2. **不改业务逻辑**：不修改 `TaskManager`、四层 Agent、物理步。
3. **时间统一**：所有表记录 `time_step`（整数逻辑步）和 `logic_time_s`（`time_step × logic_dt`，默认 `logic_dt=1.0` 秒）。`resource_event_log` 额外保留 `sim_time_s`（物理仿真时间）供对照。
4. **资源 ID 稳定**：同一物理对象在整个 run 内 `resource_id` 不变，便于窗口聚合与 GNN 建图。

## 2. 时间与 ID 约定

### 2.1 时间字段

| 字段 | 类型 | 计算方式 | 说明 |
|------|------|----------|------|
| `time_step` | int | `env["time_step"]` | 环境逻辑步计数，与 `processing_time`、subtask 计时同单位 |
| `logic_time_s` | float | `time_step × logic_dt` | 默认 `logic_dt = 1.0`，与工序/子任务时长同量纲 |
| `sim_time_s` | float | `time_step × sim.dt` | 仅 `resource_event_log`；默认 `sim.dt = 1/120` |
| `run_id` | str | 启动时生成，如 `2026-07-06_12-00-00_seed42` | 一次 `train.py` 运行唯一标识 |
| `episode_id` | int | 从 0 递增；每次 `reset_env` +1 | 当前 rule_based 无限循环，可先用 `episode_id=0` 或按「全部完工」自定义 |
| `env_id` | int | `HcSingleEnv.env_id` | 向量化环境索引 |

窗口聚合阶段以 `logic_time_s` 对齐到 10s / 30s / 60s 窗口（调研文档建议）。

### 2.2 job_id 约定

| 字段 | 取值 |
|------|------|
| `job_id` | `product_index`（`task_record["product_index"]`） |
| `product_type` | `task_record["product"]`，如 `ProductWaterPipe` |
| `task_id` | `f"{product_index}_{task}"`，如 `3_logistic_for_pipe_cutting` |

同一 `job_id` 在不同工序段用 `station_id` + `task` 区分多行 `job_trace`。

## 3. 资源 ID 命名规范

GNN 节点与 `resource_event_log.resource_id` 使用下表命名（**第一版定稿后不要改**）：

| resource_type | resource_id 格式 | 示例 |
|---------------|------------------|------|
| `machine` | `{machine_type}_ws{workstation_idx}` | `num02_rollerbedCNCPipeIntersectionCuttingMachine_ws0` |
| `gantry` | `gantry_{gantry_idx}` | `gantry_0` |
| `human` | `human_{idx}` | `human_0`（对应 `num_00_NormalHuman` 等 key 中的 idx） |
| `transport_robot` | `robot_{idx}` | `robot_0` |
| `buffer` | `storage_{class_name}_{idx:02d}` | `storage_BlackStorage_00` |
| `material` | `material_{submaterial_name}` | `material_product_00_pipe`（按 submaterial 粒度） |
| `job` | `job_{product_index}` | `job_3` |

`station_id`（用于 `job_trace`）与 machine 节点对齐：

- 加工工位：`station_id = resource_id`（machine 格式）
- 物流段：`station_id = logistic_{task}` 或 `gantry_{idx}`

## 4. 状态映射表（代码 → 文档九态）

仿真原始状态在 `env_state_action_dict` 中为字符串；写入 `resource_event_log` 前映射为调研文档中的**统一状态名**。

### 4.1 Machine 工位（`env["machine"][type]["state"][ws_idx]`）

| 代码 state | 映射状态 | reason 建议 |
|------------|----------|-------------|
| `free` | `IDLE` | — |
| `working_{task}` | `PROCESSING` | `task={task}` |
| `waiting_{task}` | `BLOCKED` | 下游未就绪，物料等待加工或输出 |
| `materialReadyFor_{task}` | `WAITING` | 物料到位，等待开工 |
| `invalid` | `STOP` | 未启用工位 |

第一版无故障注入时，`DOWN` / `REPAIR` / `SETUP` / `TOOL_CHANGE` 不出现；若 `disturbance_log` 注入故障，在注入点额外写 `DOWN` 事件。

### 4.2 Gantry（`env["machine"]["num07_gantry_group"]`）

| 代码 state | 映射状态 | reason |
|------------|----------|--------|
| `free` | `IDLE` | — |
| `working_{task}` | `PROCESSING` | 龙门架执行物流 |
| `invalid` | `STOP` | 未启用 |

subtask 为 `finding_free_gantry` 且未完成时，对**等待龙门架的 job** 记 transport waiting（见 `route_transport_task.delay_reason`），不单独给 gantry 写 `STARVED`。

### 4.3 Human（`env["human"][key]["state"]`）

| 代码 state | 映射状态 | reason |
|------------|----------|--------|
| `free` | `IDLE` | — |
| `working_{task}` | `PROCESSING` | 结合 `subtask_name` 细分 |

结合 `ongoing_task_record` 的 subtask 行（`subtasks_dict["ongoing"][0]`）：

| subtask_name | 映射状态 | reason |
|--------------|----------|--------|
| `go_to_material` / `go_to_goal_area` / `go_to_processing_machine` | `PROCESSING`（移动） | `subtask=walk` |
| `control_machine` / `control_gantry` 等 | `PROCESSING` | `subtask=operate` |
| `wait` | `STARVED` | `waiting_for_gantry_or_partner` |
| （无 ongoing_task）且 state=free | `IDLE` | — |

### 4.4 Robot（`env["robot"][key]["state"]`）

与 Human 类似：`free` → `IDLE`，`working_{task}` → `PROCESSING`；subtask `wait` → `STARVED`。

### 4.5 Buffer（`env["storage"][key]`）

| 代码 state | 映射状态 | reason |
|------------|----------|--------|
| `empty` | `EMPTY` | occupancy=0 |
| `partial` | `PARTIAL` | 0 < occupancy < capacity |
| `full` | `FULL` | occupancy = capacity |

Buffer 的 `BLOCKED`/`STARVED` 在窗口聚合阶段由上下游 machine 状态推导，第一版原始日志只记 occupancy 变化。

### 4.6 Material

| 条件 | shortage_flag | 说明 |
|------|---------------|------|
| 下道工序所需 submaterial 未就位 | `1` | 从 `task_record` + `material` 判断 |
| 正常 | `0` | — |

| 字段 | 类型 | 取值来源 | 说明 |

| `station_id` | str | §3；加工用 machine ws，物流用 `gantry_{idx}` | |
| `input_buffer_id` | str \| null | `subtasks_dict["material_start_area"]` | storage key |
| `output_buffer_id` | str \| null | `subtasks_dict["material_goal_area"]` | storage 或 machine type |
| `event` | str | 见下表 | 阶段事件类型 |
| `time_step` | int | 事件发生步 | |
| `logic_time_s` | float | `time_step × logic_dt` | |
| `queue_enter_time_step` | int \| null | 首次进入等待区 | 回填 |
| `queue_leave_time_step` | int \| null | 离开等待 | 回填 |
| `process_start_time_step` | int \| null | 开始加工/物流 | `task_record["task_start_time_step"]` 或 machine→working |
| `process_end_time_step` | int \| null | 任务完成 | `task_done=True` 时 |
| `departure_time_step` | int \| null | 离开工位 | subtask 全部完成 |

**event 枚举与触发条件**：

| event | 触发条件（对比 prev → curr） |
|-------|------------------------------|
| `job_selected` | `new_product_selected` 从 False→处理；或 material `finished_task` 仍为 `none` 且新 task_record 创建 |
| `queue_enter` | machine 状态变为 `waiting_*` 或物料进入 storage |
| `process_start` | machine/human/robot 状态变为 `working_*`，或 `task_start_time_step` 首次被设置 |
| `process_end` | `task_record["task_done"]` 变为 True，或任务从 `ongoing_task_records` 移除 |
| `departure` | 同上，或与 `process_end` 同步写入 |
| `stage_complete` | 某 product 全部 6 道工序完成，进入 `progress["finished"]` |

**主要读取路径**：

```python
ongoing = env["progress"]["ongoing_task_records"]  # dict[product_index, task_record]
progress["finished"]  # 完工列表
progress["producing"] / ["producing_indexs"]
env["material"][f"num_{idx:02d}_{product_type}"]["finished_task"]
```

**后续可算**：`avg_waiting_time_s`、`processing_time_s`、`output_rate_s`、`inter_departure_var_s`。

**读取路径**：

```python

    if tr["task_type"] != "logistic":
```

### 6.4 Collector 内部状态（不写入 env）

```python
class BottleneckDataCollector:
  _prev_resource: dict[resource_id, tuple]     # (raw, norm, subtask, reason) 签名
  _prev_buffer: dict[buffer_id, tuple]         # (occ, state, material_idx_list)
  _prev_task_by_key: dict[(job_id, task), dict]  # 按 (product_index, task) 跟踪
  _queue_enter_logged / _queue_leave_logged: set
  _transport_open: dict[task_id, dict]
```

每步 `step(env)`：

1. `time_step = env["time_step"]`
2. diff machines / humans / robots / gantry → append `resource_event_log.jsonl`
3. diff storage → append `buffer_event_log.csv`
4. diff `ongoing_task_records` → 更新 `job_trace.csv`、`route_transport_task.csv`
5. diff material submaterials → append `material_inventory_log.csv`
6. 更新 `_prev_*`

## 7. 离线派生表（不在仿真内写）

### 7.1 window_feature_table.csv

按 `window_size`（10/30/60s）对每个 `resource_id` 聚合 §5 原始表，计算调研文档中的：

- Queue：`queue_length_s`、`avg_waiting_time_s`、`occupancy_ratio_s`、`queue_growth_rate_s`
- Process：`active_pct_s`、`current_active_duration_s`、`blocked_time_s`、`starved_time_s`、`inter_departure_var_s`
- System：`upstream_blocked_ratio_s`、`downstream_starved_ratio_s`、`route_delay`、`material_shortage_propagation`

### 7.2 bottleneck_label.csv

对每个窗口节点算：

```text
bottleneck_score_s =
    0.25 * norm(queue_length_s)
  + 0.20 * norm(avg_waiting_time_s)
  + 0.25 * active_pct_s
  + 0.10 * norm(current_active_duration_s)
  + 0.10 * upstream_blocked_ratio_s
  + 0.10 * downstream_starved_ratio_s

bottleneck_node_t = argmax_s bottleneck_score_s(t)
```

连续窗口 `score > threshold` 且 `duration > min_duration` → 合并为 `bottleneck_event`。

预测标签（未来 horizon H）：

| 标签 | 来源 |
|------|------|
| `will_bottleneck` | 未来 H 内是否存在 event |
| `future_bottleneck_object_id` | 第一个/最主要 event 的 node |
| `future_bottleneck_type` | resource_type |
| `time_to_start` | `event.start - t` |
| `duration` | `event.end - event.start` |
| `root_cause_reason` | 优先 `disturbance_log`，否则规则表 |
