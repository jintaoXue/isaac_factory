# 瓶颈维度

```
文献调研：制造系统瓶颈如何定义
    ↓
定义方案：我们系统中把瓶颈定义到哪些资源和哪些观测维度上
    ↓
数据采集流程：Isaac Sim 中如何记录这些维度
    ↓
标签生成：如何从状态数据形成 bottleneck event
    ↓
后续预测：模型用什么输入，预测哪些标签
    ↓
后续可补充：严重程度、shifting、root cause、反事实验证等
```



## 参考论文定义

**限制整个系统吞吐能力的关键资源**

瓶颈来源：

- 瓶颈由资源状态产生：机器、人工、物料、buffer、运输（机器人）  

  - | 资源类型          | 包含对象                                     | 作用                         |
    | ----------------- | -------------------------------------------- | ---------------------------- |
    | `human`           | 工人、人工搬运、人工点焊、人工涂漆           | 执行人工操作或辅助搬运       |
    | `transport_robot` | 运输小车、AGV、AMR                           | 在工位和 buffer 之间搬运工件 |
    | `machine`         | 切管机、压槽机、点焊台、焊接设备、涂漆工位等 | 对工件进行加工               |
    | `buffer`          | 原材料区、半成品区、成品区                   | 存放 WIP、原料、成品         |
    | `material`        | 钢管、法兰、弯头、油漆、焊丝等               | 被加工或被消耗的物料         |

  - | 资源类型          | 主要瓶颈维度                                    | 典型瓶颈表现                          |
    | ----------------- | ----------------------------------------------- | ------------------------------------- |
    | `machine`         | process state bottleneck                        | 加工慢、故障、setup 久、active 时间高 |
    | `human`           | human process bottleneck / transport bottleneck | 人工操作慢、等待人工、人搬运慢        |
    | `transport_robot` | transport bottleneck                            | 小车不足、路径拥堵、运输延迟          |
    | `buffer`          | queue state bottleneck                          | 半成品区满、成品区满、等待时间长      |
    | `material`        | material supply bottleneck                      | 法兰/弯头缺料、钢管到料不足           |

瓶颈收集数据呈现形式

- Queue State 瓶颈：从物料堆积现象判断哪里处理不过来

  - ```
    哪里排队最多？
    哪里等待时间最长？
    哪里 WIP （已经进入生产流程，但还没有成为最终成品的工件）堆积最严重？
    ```

  - | 方法                | 定义                         |
    | ------------------- | ---------------------------- |
    | Queue Length Method | 队列最长的工位是瓶颈         |
    | Waiting Time Method | 工件等待时间最长的工位是瓶颈 |

- Process State 瓶颈

  - ```
    哪个工位最忙？
    哪个工位 active 时间最长？
    哪个工位一直处于加工/故障/维修/换型？
    哪个工位造成上下游 blocked 或 starved？
    ```

  - | 方法                          | 定义                                    |
    | ----------------------------- | --------------------------------------- |
    | Utilization Method            | 利用率最高的是瓶颈                      |
    | Active Period Percentage      | active 时间占比最高的是瓶颈             |
    | Average Active Period         | 平均连续 active 时长最长的是瓶颈        |
    | Longest Current Active Period | 当前连续 active 最久的是瞬时瓶颈        |
    | Inactive Period Method        | blocked + starved 最小的工位可能是瓶颈  |
    | Inter-departure Time Variance | 出站间隔方差最小的工位可能是瓶颈        |
    | Arrow / Turning Point         | 根据 blocked/starved 的方向判断瓶颈位置 |

  - **注意一下这个active状态定义要思考一下，把哪些状态归为active，要提前明确**

- System State 瓶颈

  - ```
    工位状态 + buffer 状态 + 上下游传播关系 + 吞吐变化
    ```

  - | 方法                  | 定义                                                 |
    | --------------------- | ---------------------------------------------------- |
    | Bottleneck Walk       | 结合 blocked/starved 和 buffer 满/空判断瓶颈方向     |
    | Bottleneck Index      | 用 utilization + 前置队列数量计算瓶颈指数            |
    | Sensitivity of System | 改善某个工位后，如果系统吞吐提升最大，则该工位是瓶颈 |



> 瓶颈预测标签表现形式：
>
> - 是否发生
> - 产生/结束时间，持续时间
> - 瓶颈位置（哪个资源），**瓶颈原因**
> - **严重程度**
> - 频率
> - **shifting**
>

## 迁移到我们的系统

需要收集的瓶颈数据，数据定义，来源

### 处理逻辑

```text
资源状态日志 + 工件流转日志 + buffer occupancy + 运输任务 + 物料库存（1.数据收集模块记录的是原始事件和状态）
    ↓
按固定窗口聚合，例如每 10s / 30s / 60s（2.窗口聚合模块把原始事件转换成 Queue / Process / System 指标，如果用GNN预测，这些指标进一步组成每个节点的 node features 和全局 graph features）
    ↓
计算 Queue / Process / System 指标
    ↓
生成 bottleneck_score （3.bottleneck_score 使用同一批窗口指标生成当前瓶颈标签）
    ↓
得到 bottleneck_node
    ↓
连续窗口合并成 bottleneck_event
```

第一版可以先把原始数据理解成：

```text

模型输入层：
把窗口聚合指标整理成每个节点的 node features、边的 edge features 和全局 graph features
```

  >

### 连续窗口聚合成瓶颈标签

**bottleneck_event-标签生成规则**

把Queue / Process / System 指标组合起来，生成可解释的初始标签。

流程：

> 1. 每个窗口算 score，对每个节点 `s`，在窗口 `t` 上计算bottleneck_score_s
>
>    ```
>    第一版可以先以 machine / station 为主计算综合分数：（后续对于不同资源对象要有不同阻塞分数计算公式）
>    bottleneck_score_s =
>        0.25 * norm(queue_length_s)
>      + 0.20 * norm(avg_waiting_time_s)
>      + 0.25 * active_pct_s
>      + 0.10 * norm(current_active_duration_s)
>      + 0.10 * upstream_blocked_ratio_s
>      + 0.10 * downstream_starved_ratio_s
>    ```
>
>    
>
> 2. 然后看是否超过定义为bottle的阈值，超过的话要记录bottleneck_event，同时也看哪个资源造就瓶颈（哪个资源产生score最高）
>
>    ```
>    bottleneck_node_t = argmax_s bottleneck_score_s(t)
>    ```
>
> 3. 把连续窗口合并成 bottleneck_event
>
>    例如
>
>    | window   | bottleneck_node | score |
>    | -------- | --------------- | ----- |
>    | 100–110s | S5              | 0.82  |
>    | 110–120s | S5              | 0.85  |
>    | 120–130s | S5              | 0.87  |
>    | 130–140s | S5              | 0.80  |
>
>    如果满足：
>
>    ```
>    score > threshold
>    and duration > min_duration
>    ```
>
>    就形成：
>
>    ```
>    bottleneck_event = {
>      node: S5,
>      start_time: 100,
>      end_time: 140,
>      duration: 40,
>      bottleneck_node,
>      ...
>    }
>    ```
>
>    
>
>    4. 生成对应预测标签
>
>    比如当前输入窗口结束时间是 `t`，预测未来 `H=120s`，看这段时间的 `bottleneck_label.csv` 记录的bottleneck_event
>
>    拿到：
>
>    | 预测标签                      | 如何得到                                                     |
>   | ----------------------------- | ------------------------------------------------------------ |
>    | `will_bottleneck`             | 未来 horizon 内是否存在 bottleneck_event                     |
>    | `future_bottleneck_object_id` | 未来 horizon 内第一个/最主要 bottleneck_event 的 node        |
>    | `future_bottleneck_type`      | 该 event 的资源类型，例如 machine / buffer / material        |
>    | `root_cause_reason`           | 优先来自 `disturbance_log`（就是在收集数据时你本身来导致这个瓶颈的扰动，比如m2故障），其次由规则判断（设定某些指标多少阈值对应哪个原因，提前拟定一份规则表） |
>    | `time_to_start`               | `event.start_time - t`                                       |
>    | `duration`                    | `event.end_time - event.start_time`                          |
>    | `throughput_loss`             | 当前 throughput 与 baseline throughput 对比得到，更适合作为**标签、严重程度指标或评估指标**，可作为后续标签（同样的仿真配置下，先跑一个无扰动 baseline，得到正常吞吐量；再跑有故障、缺料、拥堵等扰动的仿真，得到实际吞吐量。） |

### 预测标签

分成四类。

| 预测任务        | 标签                                          | 说明                                                         |
| --------------- | --------------------------------------------- | ------------------------------------------------------------ |
| 是否发生瓶颈    | `will_bottleneck`                             | 未来 horizon 内是否出现瓶颈事件                              |
| 瓶颈位置        | `future_bottleneck_object_id`                 | 未来瓶颈发生在哪个 machine / human / buffer / material / route |
| 瓶颈类型 / 原因 | `future_bottleneck_type`, `root_cause_reason` | machine_failure、material_shortage、transport_delay、buffer_full、human_delay 等 |
| 时间属性        | `time_to_start`, `duration`                   | 未来多久开始，预计持续多久                                   |

```text

| 指标名称 | 中文解释 | 主要来源 | 第一版用途 |
|---|---|---|---|
| `queue_length_s` | 节点 `s` 前等待加工或运输的工件数量，反映 WIP 堆积程度 | `buffer_snapshot.csv`、`job_trace.csv` | 节点特征；参与 `bottleneck_score` |
| `avg_waiting_time_s` | 工件进入节点 `s` 前的平均等待时间，等待越久说明越拥堵 | `job_trace.csv` | 节点特征；参与 `bottleneck_score` |
| `occupancy_ratio_s` | buffer 当前占用比例，计算为 `occupancy / capacity` | `buffer_snapshot.csv`、`episode_config.csv` | 节点特征；判断 buffer 是否接近满 |
| `queue_growth_rate_s` | 队列增长速度，表示窗口结束时队列比开始时增加了多少 | `buffer_snapshot.csv` | 节点特征；判断拥堵是否正在加剧 |
| `max_waiting_time_s` | 当前窗口内单个工件最长等待时间，反映极端等待情况 | `job_trace.csv` | 后续补充；用于异常严重程度分析 |
| `full_duration_ratio_s` | buffer 在窗口内处于满状态的时间占比 | `buffer_snapshot.csv` | 后续补充；用于 buffer bottleneck 诊断 |

| 指标名称 | 中文解释 | 主要来源 | 第一版用途 |
|---|---|---|---|
| `active_pct_s` | 节点 `s` 在窗口内处于 active 状态的时间占比。active 可包括 `PROCESSING`、`SETUP`、`DOWN`、`REPAIR` 等 | `resource_event_log.jsonl` | 节点特征；参与 `bottleneck_score` |
| `current_active_duration_s` | 截至窗口结束时，节点 `s` 已经连续 active 的时长，用于判断瞬时瓶颈 | `resource_event_log.jsonl` | 节点特征；参与 `bottleneck_score` |
| `blocked_time_s` | 节点 `s` 在窗口内因下游满或无法输出而被阻塞的时间 | `resource_event_log.jsonl`、下游 `buffer_snapshot.csv` | 节点特征；判断瓶颈可能在下游 |
| `starved_time_s` | 节点 `s` 在窗口内因上游无工件、无物料或无运输而等待的时间 | `resource_event_log.jsonl`、上游 `buffer_snapshot.csv`、`material_inventory_log.csv` | 节点特征；判断瓶颈可能在上游 |
| `inter_departure_var_s` | 节点 `s` 相邻出站时间间隔的方差，反映输出节奏是否稳定 | `job_trace.csv` | 节点特征；作为 process state 辅助指标 |
| `processing_time_s` | 节点 `s` 在窗口内实际加工时间总和 | `resource_event_log.jsonl`、`job_trace.csv` | 后续补充；可用于计算利用率 |
| `setup_time_s` | 节点 `s` 在窗口内装夹、换型、准备、调参的时间 | `resource_event_log.jsonl` | 后续补充；用于分析 setup 瓶颈 |
| `down_time_s` | 节点 `s` 在窗口内故障停机时间 | `resource_event_log.jsonl`、`disturbance_log.csv` | 后续补充；用于判断 machine failure |
| `repair_time_s` | 节点 `s` 在窗口内维修时间 | `resource_event_log.jsonl` | 后续补充；用于维护维修瓶颈 |
| `active_time_s` | 节点 `s` 在窗口内所有 active 状态的累计时间 | `resource_event_log.jsonl` | 中间指标；用于计算 `active_pct_s` |

## 预测输入数据的节点特征解释与来源表

GNN 输入中的节点特征不是单独重新采集的，而是由上面的窗口聚合指标整理得到的。每个时间窗口会生成一个 production graph snapshot，节点包括 `machine`、`human`、`transport_robot`、`buffer`、`material`。不同类型节点可使用同一套特征向量，不适用的字段填 0，并额外加入 `resource_type` 标识。

| 节点特征 | 中文解释 | 主要来源 | 适用节点类型 | 备注 |
|---|---|---|---|---|
| `queue_length` | 当前节点相关 buffer 或工位前的等待工件数量 | `buffer_snapshot.csv`、`job_trace.csv` | `machine`、`buffer` | 对 `machine` 可取其 input buffer 的队列长度 |
| `avg_waiting_time` | 当前窗口内工件进入该节点前的平均等待时间 | `job_trace.csv` | `machine`、`buffer`、`transport_robot` | 运输节点可用 transport waiting time 替代 |
| `active_pct` | 当前节点在窗口内处于 active 状态的时间占比 | `resource_event_log.jsonl` | `machine`、`human`、`transport_robot` | 对 `buffer` 和 `material` 可填 0 |
| `current_active_duration` | 当前节点截至窗口结束时连续 active 的时长 | `resource_event_log.jsonl` | `machine`、`human`、`transport_robot` | 用于捕捉 momentary bottleneck |
| `blocked_ratio` | 当前节点在窗口内处于 blocked 状态的时间比例 | `resource_event_log.jsonl` | `machine`、`human`、`transport_robot`、`buffer` | 对工位表示下游满；对路线表示通行受阻 |
| `starved_ratio` | 当前节点在窗口内处于 starved / waiting 状态的时间比例 | `resource_event_log.jsonl`、`material_inventory_log.csv` | `machine`、`human`、`transport_robot` | 反映上游无工件、缺料或等待运输 |
| `transport_waiting_time` | 当前节点相关运输任务的平均等待时间 | `route_transport_task.csv` | `transport_robot`、`machine`、`buffer` | 对机器可表示等待小车来取/送的时间 |
| `material_shortage_flag` | 当前节点是否受到物料短缺影响 | `material_inventory_log.csv`、`resource_event_log.jsonl` | `material`、`machine` | 对非相关节点填 0 |
| `occupancy_ratio` | 当前 buffer 或物料库存占容量比例 | `buffer_snapshot.csv`、`material_inventory_log.csv` | `buffer`、`material` | 对 machine 可使用 input buffer occupancy |
| `output_rate` | 当前窗口内该节点完成加工或输出的数量 / 窗口长度 | `job_trace.csv` | `machine`、`buffer` | 用于反映节点实际产出能力 |

## 预测输入数据的全局特征解释与来源表

全局特征描述的是整个生产系统在当前窗口或过去一段时间内的状态，不属于某一个单独节点。它通常由 `episode_config.csv`、所有节点窗口特征汇总、成品出站记录和扰动日志共同计算得到。

| 全局特征 | 中文解释 | 获取方式 / 来源 | 是否建议第一版使用 |
|---|---|---|---|
| `total_WIP` | 当前系统内所有在制品数量，包括 buffer 中等待的工件和正在加工的工件 | 所有 `buffer_snapshot.csv` 的 occupancy 加总 + 正在加工 job 数量 | 建议使用 |
| `throughput_rolling` | 最近一段时间的滚动吞吐率，例如过去 60s 成品输出数量 / 60s | 成品区 `job_trace.csv` 或 finished buffer 出站记录 | 建议使用 |
| `arrival_rate` | 当前 episode 的订单/工件到达率 | 优先来自 `episode_config.csv`；也可由窗口内新 job 数量统计 | 建议使用 |
| `product_mix` | 当前生产的产品类型比例或任务类型组合 | `episode_config.csv` 或窗口内 `job_trace.csv` 的 `product_type` 统计 | 有多产品类型时使用 |
| `disturbance_flag` | 当前窗口是否存在已经发生的扰动，例如故障、缺料、路线阻塞、人工不可用 | 查询 `disturbance_log.csv`：若窗口与扰动时间段重叠则为 1 | 建议使用，但只能使用当前或历史已发生扰动 |
| `shift_time` | 最近一次已发生瓶颈迁移的时间，或距离上次迁移的时间 | 由历史 `bottleneck_event` 序列计算 | 后续使用；第一版不建议直接用未来 shift 信息 |
| `system_utilization_mean` | 全部 machine 节点 active_pct 的平均值，表示系统整体忙碌程度 | 对所有 machine 节点的 `active_pct` 求平均 | 可选 |
| `num_blocked_nodes` | 当前窗口中处于 blocked 状态的节点数量 | 对所有节点的 `blocked_ratio` 统计 | 可选 |
| `num_starved_nodes` | 当前窗口中处于 starved 状态的节点数量 | 对所有节点的 `starved_ratio` 统计 | 可选 |
| `current_output_rate` | 当前窗口最终成品产出速度 | 成品 buffer 或最后一道工序的 departure 记录 | 可选；与 `throughput_rolling` 类似 |
