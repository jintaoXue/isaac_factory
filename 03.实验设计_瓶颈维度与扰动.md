# 实验设计：四维瓶颈与扰动注入

> 目标：当前 hc_factory 是**确定性正常生产**环境（固定工序时间、固定资源数量、无故障）。  
> 通过在 **物料 / 工人 / 物流运输 / 工位** 四个维度改参并注入扰动，丰富运行轨迹，提升瓶颈识别与诊断模型的训练效果。

## 0. 怎么跑（CLI 选维度）

**每次运行用参数决定注入哪一维**；默认 `none` = 不注入（正常生产）。

```bash
conda activate env_isaaclab
python train.py \
  --task HRTPaHC-v1 \
  --algo rule_based \
  --num_envs 1 \
  --seed 42 \
  --device cuda:0 \
  --headless \
  --disturbance_dim machine \
  --disturbance_intensity 1.0
```

| 参数 | 取值 | 说明 |
|------|------|------|
| `--disturbance_dim` | `none` / `material` / `human` / `logistics` / `machine` | **主扰动维度**（单次只开一维） |
| `--disturbance_intensity` | float ≥ 0，默认 `1.0` | 强度：放大噪声、缺料比例、事件时长 |
| `--disturbance_human_count` | int，可选 | 仅 `human`：覆盖工人数 |
| `--disturbance_agv_count` | int，可选 | 仅 `logistics`：覆盖 AGV 数 |
| `--disturbance_gantry_count` | int，可选 | 仅 `logistics`：覆盖龙门架数 |

对照示例：

```bash
# 基线
--disturbance_dim none

# 物料短缺 + 不良返工
--disturbance_dim material --disturbance_intensity 1.0

# 减人 + 操作变慢；可覆盖人数
--disturbance_dim human --disturbance_intensity 1.0
--disturbance_dim human --disturbance_human_count 3

# 减 AGV / 龙门架 + 运输变慢
--disturbance_dim logistics --disturbance_intensity 1.0
--disturbance_dim logistics --disturbance_agv_count 1 --disturbance_gantry_count 1

# 加工时间噪声 + 成功率 + 中段工位故障
--disturbance_dim machine --disturbance_intensity 1.0
```

日志：`episode_config.csv` 含 `disturbance_dim` / `disturbance_intensity` / `disturbance_applied`；事件写入 `disturbance_log.csv`。

## 1. 瓶颈定义（本项目操作化）

**瓶颈**：在任意观察窗口内，**限制整条产线吞吐（throughput）的资源或工序**。

本环境中，瓶颈由下列四类因素产生；四者**相互关联、非独立**：

| 维度 | 英文标签 | 典型可调参数 |
|------|----------|--------------|
| 物料 | `material` | 短缺、损坏/不良 |
| 工人 | `human` | 数量、效率、熟练度、疲劳 |
| 物流运输 | `logistics` | AGV 数量、龙门架数量、初始位置 |
| 工位 | `machine` | 是否故障、是否顺利完成加工 |

> **关联性示例**：物料严重短缺时，产线整体闲置，机器故障或工人疲劳即使发生，也很难表现为「系统级吞吐瓶颈」——缺料会**掩盖**其它维度的瓶颈信号。因此实验设计必须控制「主扰动维度」，并记录其它维度是否被压制。

## 2. 总体建议：怎么往确定环境里注入扰动

### 2.1 原则（推荐）

1. **先有负载，再注扰动**  
   订单过少（如默认 5）时系统几乎不堵，扰动效果弱。建议在 G1 级负载（订单 ≥10–15，并行上限仍可保持 5）上再做扰动实验。

2. **参数围绕标称值随机浮动，而不是写死另一组常数**  
   每道工序的操作时间、成功率等，设为「标称值 ± 噪声」：
   ```text
   T_eff  = max(T_min,  T_nominal + ε),   ε ~ N(0, σ_t²) 或 Uniform(-a, a)
   p_ok   = clip(p_nominal + δ, 0, 1),    δ ~ N(0, σ_p²)
   ```
   这样同一配置跑多次仍有多样性，又不会偏离「正常生产」太远。

3. **单因子优先，再做组合**  
   每个 run **主注入一个维度**（由 `--disturbance_dim` 指定），便于用 `disturbance_log` 做有监督根因标签。

4. **分层实现**  
   | 层级 | 含义 | 本实现 |
   |------|------|--------|
   | **L0 配置级** | episode 开始前改 cfg | 人/AGV/龙门架数量（`apply_disturbance_to_cfgs`） |
   | **L1 采样级** | 加工/人工时间噪声、成功率 | `machine.py` / `human.py` 读 `RuntimeDisturbanceCfg` |
   | **L2 运行时事件** | 中途故障/离岗/龙门架不可用 | `DisturbanceInjector` |

## 3. 各维度实际注入内容（intensity=1.0 时）

| 维度 | L0 | L1 | L2 事件（约） |
|------|----|----|----------------|
| `material` | — | 跳过部分法兰/弯头入库（缺料）；加工成功率 ≈0.9，失败则返工 | 无 |
| `human` | 工人 5→3（可 `--disturbance_human_count` 覆盖） | 人工时间 ×1.35 + 更大噪声 | ~600 步起，一名空闲工人离岗约 150 步 |
| `logistics` | AGV→1，龙门架→1（可覆盖） | 龙门架动画变慢 + 噪声增大 | ~700 步起，龙门架 0 不可用约 100 步 |
| `machine` | — | 加工时间 σ≈5；成功率 ≈0.92 | ~800 步起，切管机一工位 `invalid` 约 120 步 |
| `none` | 不改 | 保持原默认（人工 subtask 仍有轻度噪声 std=2） | 无 |

实现文件：

- `env_asset_cfg/cfg_disturbance.py` — CLI → 全局 `RuntimeDisturbanceCfg` + 改注册数量
- `src/disturbance.py` — L2 Injector + 时间/成功率采样辅助函数
- `train.py` — `--disturbance_*` 参数，在 `gym.make` 前 `apply_disturbance_to_cfgs()`

## 4. 维度关联与实验控制

| 主扰动维度 | 建议同时保持 | 避免同时强开 | 原因 |
|------------|--------------|--------------|------|
| material | 人/车/机标称 | 严重 machine DOWN | 缺料会掩盖故障信号 |
| human | 物料充足、物流标称 | 极端缺料 | 否则人闲置，效率扰动看不见 |
| logistics | 物料充足、机无故障 | 订单过低 | 无负载则争抢不明显 |
| machine | 物料充足 | 极端缺人+缺车 | 故障传播路径被其它饥饿切断 |

每个 run 标签：`primary_dimension = --disturbance_dim`（已写入 `episode_config`）。

## 5. 推荐实验路线

### Phase 0 — 正常生产基线

```bash
--disturbance_dim none --seed 42
```

### Phase 1 — 单维扰动扫描（固定负载与 seed 组）

```text
dim ∈ {material, human, logistics, machine}
intensity ∈ {0.5, 1.0, 1.5}
seed ∈ {42, 43, 44}
```

### Phase 2 — 资源覆盖细扫（可选）

在 `human` / `logistics` 下用 `--disturbance_human_count` 等做离散水平对照。

## 6. 评价指标

| 层级 | 看什么 |
|------|--------|
| 系统 | 完工时间、吞吐、总 WIP |
| 节点 | `active_pct`、`starved_time`、`blocked_time`、队列长度 |
| 事件 | `bottleneck_event` 是否随 `disturbance_dim` 变化 |
| 诊断 | `disturbance_log` 为真值时的 `root_cause` / `primary_dimension` 准确率 |
| 关联性 | 强 `material` 下 machine 类瓶颈占比应下降 |
