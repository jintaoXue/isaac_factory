## 1. 文献中的瓶颈维度框架

### 1.1 什么是制造瓶颈

制造文献中，**吞吐瓶颈（throughput bottleneck）** 通常指：在一段观察窗口内，**限制整条产线或车间产出速率的关键资源或工序**。改善该资源后，系统吞吐提升最明显；反之，非瓶颈资源的局部优化对系统产出帮助有限（TOC / 约束理论的经典直觉）。

### 1.2 按「观测信息类型」分类（Rosén et al., 2023）

系统综述将瓶颈检测方法按所用信息分为三类：

| 维度 | 核心问题 | 典型指标 | 文献代表方法 |
|------|----------|----------|--------------|
| **Queue State** | 哪里在排队？ | 队列长度、等待时间、WIP、buffer 占用率 | Queue Length Method、Waiting Time Method |
| **Process State** | 哪里最忙/最久活跃？ | 利用率、active 时间占比、连续加工时长、blocked/starved 时间 | Utilization、Active Period Percentage、Longest Current Active Period |
| **System State** | 瓶颈如何沿工艺传播？ | 上下游 blocked/starved 比例、路线延迟、吞吐损失、敏感性 | Bottleneck Walk、Bottleneck Index、Sensitivity of System |

> **对本项目的含义**：采集层应同时保留 **队列类原始事件**（buffer、job 等待）和 **过程类原始事件**（资源状态变化），离线阶段再合成 System 指标。这与 `瓶颈数据采集规范.md` 中 7 表设计一致。

### 1.3 按「管理任务」分类（Vicentin et al., 2021）

AI 与数据驱动瓶颈研究常把任务分为四阶段：

| 阶段 | 目标 | 本项目对应 |
|------|------|------------|
| **Identify** | 判断「谁是瓶颈」 | `bottleneck_node`、`bottleneck_score` |
| **Diagnose** | 判断「为什么堵」 | `root_cause_reason`、与 `disturbance_log` 对齐 |
| **Predict** | 预测未来是否/何时堵 | `will_bottleneck`、`time_to_start` |
| **Prescribe** | 给出调度/扩容建议 | 后续工作，不在第一版采集范围 |

### 1.4 按「资源类型」分类（本项目 + 文献综合）

| 资源类型 | 典型瓶颈表现 | 文献/工业常见根因 |
|----------|--------------|-------------------|
| `machine` | 加工慢、工位忙、故障停机 | 产能不足、换型/setup、微停、设备故障 |
| `human` | 操作慢、等待协同、走行时间长 | 人手不足、技能差异、等待龙门架/机器 |
| `transport_robot` / 龙门架 | 搬运延迟、等车、路径慢 | AGV/龙门架数量不足、路径拥堵 |
| `buffer` | 满库阻塞上游、空库饿死下游 | 有限 buffer 导致 blocking/starvation |
| `material` | 缺料、齐套失败 | 供应链波动、多子件未齐套 |

### 1.5 Blocking 与 Starvation（DES 经典概念）

在有限 buffer 的产线模型中（Hopp & Spearman；大量 DES 文献）：

- **Starvation（饥饿）**：工位因**无输入**（空 buffer、缺料、无运输）而空闲
- **Blocking（阻塞）**：工位因**无法向下游输出**（下游满、下游未就绪）而被迫等待

二者是传播型瓶颈的核心机制：上游堆积往往由下游瓶颈引起，下游空闲往往由上游供料不足引起。

## 2. 本环境的瓶颈维度定义

结合文献与 `瓶颈维度调研0629_tyx.md`，建议在 hc_factory 中把「瓶颈」操作化为 **三维交叉**：

```
瓶颈 = 资源节点 s × 观测维度 d × 根因类型 r
```

### 2.1 资源节点 s（GNN 节点）

| resource_type | 代码/日志 ID 示例 | 说明 |
|---------------|-------------------|------|
| `machine` | `num02_..._ws0`、`gantry_0` | 加工工位 + 龙门架 |
| `human` | `human_0` | 工人 |
| `transport_robot` | `robot_0` | AGV |
| `buffer` | `storage_BlackStorage_00` | 原料/半成品/成品仓 |
| `material` | `material_product_00_pipe` | 子物料粒度 |

### 2.2 观测维度 d（窗口聚合指标）

| 文献类别 | 第一版指标（窗口内） | 主要数据来源 |
|----------|---------------------|--------------|
| Queue | `queue_length_s`、`avg_waiting_time_s`、`occupancy_ratio_s` | `buffer_event_log`、`job_trace` |
| Process | `active_pct_s`、`current_active_duration_s`、`blocked_time_s`、`starved_time_s` | `resource_event_log` |
| System | `upstream_blocked_ratio_s`、`downstream_starved_ratio_s`、`route_delay`、`throughput_loss` | 多表联合 + baseline 对比 |

### 2.3 统一状态九态（与采集规范对齐）

原始字符串状态映射为：

`IDLE` / `PROCESSING` / `BLOCKED` / `STARVED` / `WAITING` / `DOWN` / `REPAIR` / `SETUP` / `STOP`

第一版无故障注入时，常见的是：`PROCESSING`、`BLOCKED`、`STARVED`、`WAITING`、`IDLE`。

### 2.4 根因类型 r（用于 diagnose / root_cause）

建议第一版至少区分：

| root_cause | 含义 |
|------------|------|
| `capacity_limit` | 资源数量或工位数不足（结构性瓶颈） |
| `process_time_long` | 加工/subtask 耗时长 |
| `transport_delay` | 物流/龙门架/AGV/人工走行慢 |
| `buffer_full` | 下游或中间 buffer 满导致 blocking |
| `material_shortage` | 子物料未齐套或缺料 |
| `partner_wait` | 多智能体 subtask 协同等待（如 human `wait` 等龙门架） |
| `scheduling_suboptimal` | 调度策略导致非最优资源占用 |
| `machine_failure` | 注入故障（扰动实验） |
| `storage_exhausted` | 找不到空闲仓位（`_find_free_storage` 失败） |

## 3. hc_factory 中可能造成瓶颈的原因

以下按 **是否已在当前 rule_based 运行中自然出现** 与 **代码位置** 整理。

### 3.1 工艺与产能结构（结构性瓶颈）

水喉（`ProductWaterPipe`）工艺链为：

```
物流 → 切管 → 物流 → 压槽 → 物流 → 点焊 → 物流 → 氩弧焊 → 物流 → MIG → 物流 → 涂漆
```

每一道加工前几乎都有 **物流 task**，使 **龙门架 + 人 + AGV** 与 **机器** 形成串联竞争。

| 原因 | 机制 | 代码/配置 | 当前默认值 |
|------|------|-----------|------------|
| 机器工位有限 | 并行加工受工位数约束 | `cfg_machine.py` `num_workstations` | 多数机器 1–2 工位 |
| 龙门架数量少 | 多 job 争抢 `finding_free_gantry` | `active_gantry_indices` | `[0, 1]` 仅 2 台 |
| AGV 数量少 | `have_AGV` 物流链依赖 robot | `CfgRobotRegistrationInfos` | 2 台 |
| 工人数量少 | 人工 subtask 串行、Agent D 选第一个空闲人 | `CfgHumanRegistrationInfos` | 5 人 |
| 并行在制品上限 | 同时生产的批次受限 | `single_env_parallel_producing_limit` | 5 |
| 订单量小 | 系统负载低，瓶颈不明显 | `CfgProductOrder` | 5 根水管 |

**实验含义**：在不做任何扰动时，系统已可能存在 **龙门架或热点机器（如切管机、焊接机器人）** 的结构性瓶颈，但 5 件订单 + rule_based 固定规则下，瓶颈可能 **弱且重复**。

### 3.2 物流与运输（Transport bottleneck）

| 原因 | 表现 | 代码线索 |
|------|------|----------|
| 等空闲龙门架 | job 在 subtask `finding_free_gantry` 停留 | `task_progress_manager.py`、`route_transport_task.delay_reason=finding_free_gantry` |
| 人工走行距离 | 工人 `go_to_material` / `go_to_goal_area` 路径步数不同 | `human.py` + `route.py`；reset 时 `torch.randperm` 分配初始点 |
| 人机协同等待 | human subtask 为 `wait`，映射为 `STARVED` | `cfg_process_subtask_gallery.py` |
| AGV 与无 AGV 分支 | 有/无 robot 时 subtask 序列不同 | `have_AGV` vs `only_have_gantry` |
| 龙门架动画时间固定 | `animation_time: 20` | `cfg_machine.py` num07_gantry_group |

### 3.3 Buffer 与 WIP（Queue state bottleneck）

| 原因 | 表现 | 代码线索 |
|------|------|----------|
| Black/Yellow 仓容量小 | `capacity: 6`，易 `full` | `cfg_storage.py` |
| 半成品需占位 | 加工完成后 `_find_free_storage` 找黄色/黑色仓 | `task_progress_manager._find_free_storage` |
| 仓满无法落料 | 抛出 `No free storage found` | 高负载时的硬瓶颈 |
| 机器 `waiting_{task}` | 下游未就绪，工位等料 | `machine.state` → `BLOCKED` |
| 机器 `materialReadyFor_{task}` | 料已到，等开工 | `WAITING` |

### 3.4 物料与齐套（Material bottleneck）

| 原因 | 表现 | 代码线索 |
|------|------|----------|
| 多子件工序 | `batch_spot_welding` 需 pipe + flange + elbow | `CfgProductProcessGallery` |
| 子料初始在固定库区 | 原料在 Black/Ground storage 占位 | `material.reset_raw_materials_to_storage` |
| 在制品/disappear 状态 | 未进入工序的料不可见 | `material_states_in_subtasks` |

### 3.5 加工时间（Process time bottleneck）

| 原因 | 表现 | 代码线索 |
|------|------|----------|
| 固定 subtask 计时 | 人/机操作按步数倒计时完成 | `CfgSubtaskPredefinedTimeGallery` |
| 配置中的加工时间 | `process_time: 20` 每道工序 | `CfgProductProcessGallery` |
| 随机加工时间（未接入） | `gaussian_random_time: 10` 仅写入 `episode_config` | **尚未在 runtime 使用** |
| subtask 噪声（计划中） | `coding_note.md` §7.5 | **尚未实现** |

### 3.6 调度与决策（Scheduling-induced bottleneck）

当前 `rule_based` **不是随机规划**，而是确定性规则：

| Agent | 规则 | 可能后果 |
|-------|------|----------|
| A | 选 mask 中第一个产品 | 产品顺序固定 |
| B | 选第一个可生产批次 | 可能饿死其他在制品 |
| C | 选 mask 中**最后一个**合法 task | 偏向工艺链后端 task |
| D | 选第一个空闲人/AGV | 负载不均，某些人永远更忙 |

这会造成 **可重复的调度型瓶颈**（例如总是同一人、同一龙门架先被占用），但不是「每次规划不同」带来的随机性。

### 3.7 初始化随机（Episodic variability）

| 来源 | 影响 |
|------|------|
| `human.py` / `robot.py` reset 时 `torch.randperm` | 人/AGV 初始地图点不同 → 首段走行时间不同 |
| seed 未完全绑定 randperm | 不同 run 间可能不可复现（见 ARCHITECTURE 讨论） |

## 4. 建议的实验标签体系（第一版）

### 4.1 瓶颈事件（bottleneck_event）

在固定窗口（建议 10s / 30s / 60s）内，对每个资源节点计算 `bottleneck_score`（见 `瓶颈维度调研0629_tyx.md`），超过阈值且持续 `min_duration` 则合并为事件：

```text
bottleneck_event = {
  node_id,           # 如 num02_..._ws0 / gantry_0 / storage_BlackStorage_01
  resource_type,     # machine / human / transport_robot / buffer / material
  observation_dim,   # queue / process / system（主因维度）
  root_cause,        # §2.4
  start_time, end_time, duration,
  severity,          # 可选：throughput_loss 或 score 峰值
}
```

### 4.2 预测任务（与调研文档一致）

| 标签 | 说明 |
|------|------|
| `will_bottleneck` | 未来 H 秒内是否发生瓶颈事件 |
| `future_bottleneck_object_id` | 未来瓶颈节点 |
| `future_bottleneck_type` | 资源类型 |
| `root_cause_reason` | 优先来自 `disturbance_log`，否则规则推断 |
| `time_to_start` / `duration` | 时间属性 |

### 4.3 建议的「根因—指标」规则表（离线）

| root_cause | 主要证据指标 |
|------------|--------------|
| `transport_delay` | `route_transport_task.delay_reason`、gantry/robot `starved_ratio` 高 |
| `buffer_full` | buffer `FULL` 占比高 + 上游 machine `BLOCKED` |
| `material_shortage` | `material_inventory_log.shortage_flag` |
| `capacity_limit` | 某 machine `active_pct` 持续高 + 无注入故障 |
| `partner_wait` | human `STARVED` + subtask `wait` |
| `machine_failure` | `disturbance_type=machine_failure` 时间重叠 |
| `storage_exhausted` | 仿真异常或 buffer 全满持续 |

## 5. 可注入扰动清单

扰动按 **实现成本** 分层：A 仅改配置即可；B 需少量逻辑；C 需新模块但已有日志字段预留。

### 5.1 A 类：Episode 级配置扰动（改 cfg / yaml，无需 disturbance 模块）

适合 **因子实验**：不同 run 用不同配置，写入 `episode_config.csv`。

| 扰动 ID | 操作 | 预期瓶颈类型 | 主要修改位置 |
|---------|------|--------------|--------------|
| `P01_order_scale` | 订单 5→10→20 | `capacity_limit`、buffer 压力 | `CfgProductOrder`、`CfgRegistrationInfos`、`parallel_producing_limit` |
| `P02_human_scale` | 工人 5→8→10 | `capacity_limit`（人工） | `CfgHumanRegistrationInfos` |
| `P03_agv_scale` | AGV 1→2→N | `transport_delay` 缓解或转移 | `CfgRobotRegistrationInfos`、`robot_upper_bound`、路线 init 点 |
| `P04_gantry_scale` | 启用更多龙门架 index | `finding_free_gantry` 减少 | `active_gantry_indices`（USD 有 4 台，默认开 2 台） |
| `P05_buffer_tight` | Black/Yellow capacity 6→2 | `buffer_full`、blocking | `cfg_storage.py` |
| `P06_buffer_loose` | capacity 6→20 | 减轻 blocking | 同上 |
| `P07_process_slow` | `process_time` 20→40 | `process_time_long` | `CfgProductProcessGallery` |
| `P08_subtask_slow` | 人/机 subtask 步数加倍 | `process_time_long` | `CfgSubtaskPredefinedTimeGallery` |
| `P09_parallel_wip` | `parallel_producing_limit` 3 vs 8 | 系统 WIP 与争抢 | `cfg_hc_env.py`、`marl.yaml` |
| `P10_init_scatter` | 固定/随机 seed 控制人/AGV 初始点 | `transport_delay` 方差 | `human.py`/`robot.py` reset + `torch.manual_seed` |
| `P11_algo_policy` | rule_based vs MARL | `scheduling_suboptimal` | `--algo marl` vs `rule_based` |

### 5.2 B 类：Runtime 参数扰动（单 episode 内可时变，建议写 `disturbance_log`）

需在 `step_env_logic` 前/中增加 **DisturbanceInjector**（读表或调度脚本），但不必改物理步。

| 扰动 ID | 注入方式 | disturbance_type | 预期效果 |
|---------|----------|------------------|----------|
| `P20_machine_down` | 指定工位 `state=DOWN` 持续 T 步 | `machine_failure` | 下游 starvation + 上游 blocking |
| `P21_machine_slow` | 临时增大该工位 processing 计数 | `process_time_increase` | 局部 active 时间变长 |
| `P22_human_absent` | 将 `human_k` 从 mask 剔除 T 步 | `human_unavailable` | 人工 subtask 饥饿 |
| `P23_agv_absent` | 同上，robot | `human_unavailable` / `transport_delay` | 物流切到 only_have_gantry 或等待 |
| `P24_gantry_down` | 某 `gantry_idx` 强制 non-free | `machine_failure` | `finding_free_gantry` 恶化 |
| `P25_material_hold` | 延迟某 submaterial 入库 | `material_shortage` | 齐套工序卡住 |
| `P26_storage_cap` | 运行时改某 storage `capacity` | `buffer_full` | 突发满库 |
| `P27_order_surge` | 中途增加 `not_started` 计数 | `arrival_surge` | WIP 冲击 |
| `P28_route_slow` | 全局或局部放大 transport 步数 | `transport_delay` | route 模块乘系数 |
| `P29_partner_delay` | 延长 gantry `animation_time` | `transport_delay` | 协同 wait 变长 |

### 5.3 C 类：已实现配置但尚未生效的扰动（优先开发）

| 扰动 | 状态 | 说明 |
|------|------|------|
| `gaussian_random_time` | 仅在 `episode_config` 记录 | 应在 machine 加工计时处采样 N(0, σ) |
| subtask 时间噪声 | `coding_note.md` §7.5 待做 | 对 `CfgSubtaskPredefinedTimeGallery` 加噪声 |
| `disturbance_log.csv` 写入 | 表结构已有，运行时为空 | 注入模块落地后即可做有真值 root cause 的监督 |

### 5.4 D 类：自然发生、无需注入但需控制的「软扰动」

| 现象 | 是否算扰动 | 实验建议 |
|------|------------|----------|
| reset 后人/AGV 位置随机 | 是（初始化噪声） | 固定 seed 做可复现；换 seed 做多样性 |
| 多 job 争抢龙门架 | 否（系统内生） | 增大 `P01`+`P04` 订单/并行度以放大 |
| rule_based 固定决策 | 否（策略确定） | 作 baseline；换 MARL 作对照 |
| 物理步 timing | 与逻辑瓶颈无关 | 采集瓶颈数据可 `logic_only` 跳过物理步 |

### 5.5 disturbance_log 字段（与采集规范一致）

注入任何 B/C 类扰动时建议写入：

```text
disturbance_type, target_resource_id, start_time_step, end_time_step,
severity, parameter_before, parameter_after, notes
```

`root_cause_reason` 标签应 **优先与 disturbance 对齐**，这是有监督诊断实验的金标准。

## 6. 实验设计建议

### 6.1 三类 baseline

| 实验组 | 目的 |
|--------|------|
| **G0 结构性 baseline** | 当前默认配置 + rule_based，观察「无注入」下的弱瓶颈 |
| **G1 负载扫描** | 固定策略，扫描 `P01`×`P02`×`P04`（订单×人数×龙门架） |
| **G2 扰动对照** | 固定负载，单次只注入一种 `P20`–`P29`，有 `disturbance_log` 真值 |

### 6.2 推荐最小因子设计（第一版论文/试点）

```text
订单量:     {5, 10, 20}
并行上限:   {5, 10}
龙门架数:     {1, 2}
工人数:     {5, 8}
seed:       {42, 43, 44}   # 每个组合 3 次重复

→ 主效应：负载是否足以产生可检测瓶颈
→ 交互：运输资源 vs 机器产能
```

### 6.3 数据采集设置

- 采集挂点：`HcSingleEnvBase.step_env_logic` 末尾（已实现 `BottleneckDataCollector`）

- 每个 run 保留 `episode_config.csv` + 7 张原始表

### 6.4 评价指标

| 层级 | 指标 |
|------|------|
| 系统 | 完工时间、吞吐（件/逻辑步）、总 WIP |
| 节点 | `active_pct`、`starved_time`、`blocked_time`、队列长度 |
| 预测 | `will_bottleneck` AUROC、提前量 `time_to_start` MAE |
| 诊断 | `root_cause_reason` 分类准确率（仅 G2 有真值） |

### 6.5 当前环境的局限（设计实验时需知晓）

1. **仅一种产品**（`ProductWaterPipe`），`product_mix` 因子尚无效  
2. **rule_based 确定性**，若不注入扰动或提高负载，轨迹多样性不足  
3. **gaussian_random_time / subtask 噪声未接入**，加工时间方差因子暂不可用  
4. **AGV/人/物料受 USD prim 数量上限**，扩规模见 ARCHITECTURE_CN §运行方式  
5. **disturbance 注入器未实现**，G2 类实验需先开发小型 Injector
