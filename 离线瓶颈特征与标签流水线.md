## 3. 如何运行

在仓库根目录或任意目录：

```bash
python3 source/isaaclab_tasks/isaaclab_tasks/direct/hc_factory/tools/build_bottleneck_features.py \
  --run_dir source/isaaclab_tasks/isaaclab_tasks/direct/hc_factory/output/bottleneck_dataset/2026-07-18_17-46-35_seed42 \
  --window_sizes 30,60 \
  --horizon 60 \
  --score_threshold 0.55 \
  --min_event_windows 2
```

| 参数 | 默认 | 含义 |
|------|------|------|
| `--run_dir` | 必填 | 含 `env_XX/` 的 run 目录 |
| `--window_sizes` | `30,60` | 逻辑秒窗口（`logic_dt=1` 时等于步数） |
| `--horizon` | `60` | 预测标签未来视野 H（逻辑秒） |
| `--score_threshold` | `0.55` | 判定「当前窗口是瓶颈窗口」的分数阈值 |
| `--min_event_windows` | `2` | 合并为 `bottleneck_event` 所需的最少连续高分窗口数 |
| `--out_dir` | `<run_dir>/derived` | 输出根目录 |

## 4. 算法说明

### 4.1 资源状态时间轴

从 `resource_event_log.jsonl` 按 `resource_id` 排序，把离散事件展开为连续区间：

```text
[t_i, t_{i+1}) → to_state
```

窗口内对状态区间求交，得到：

- `active_pct_s`：处于 `PROCESSING` 的时间占比
- `blocked_time_s`：处于 `BLOCKED` 的时长
- `starved_time_s`：处于 `STARVED` / `WAITING` 的时长
- `current_active_duration_s`：窗口结束时刻向前连续 PROCESSING 时长

### 4.2 队列与等待

- 用 `job_trace` 的 `queue_enter` / `queue_leave` 配对，按 `station_id` 挂到对应资源
- `avg_waiting_time_s`：与窗口相交的等待时段长度均值
- `queue_length_s`：窗口内活跃等待段数量；buffer 资源则用占用均值

### 4.3 Buffer / 物流 / 缺料

- Buffer：`occupancy_ratio_s`、`queue_growth_rate_s`（窗口起止占用差 / 窗长）
- `route_delay_s`：窗口内完成的物流任务 `(end - request)` 均值
- `material_shortage_propagation_s`：窗口内 `shortage_flag=1` 行占比

### 4.4 上下游耦合（第一版启发式）

- `upstream_blocked_ratio_s` / `downstream_starved_ratio_s`：全局 BLOCKED/STARVED 占比，并对 machine/gantry 叠加自身占比（上限 1）
- 后续可改为显式工艺图邻接；当前足够跑通管道

### 4.5 瓶颈分数（规范 §7.2）

同一窗口内跨资源 min-max 归一化后：

```text
bottleneck_score_s =
    0.25 * norm(queue_length_s)
  + 0.20 * norm(avg_waiting_time_s)
  + 0.25 * active_pct_s
  + 0.10 * norm(current_active_duration_s)
  + 0.10 * upstream_blocked_ratio_s
  + 0.10 * downstream_starved_ratio_s
```

`bottleneck_node_t = argmax_s score_s(t)`。

### 4.6 事件与预测标签

- 连续 ≥ `min_event_windows` 个窗口满足 `score_max ≥ threshold` **且节点相同** → 一条 `bottleneck_event`
- 对每个窗口 t、视野 H：
  - `will_bottleneck`：`(t, t+H]` 内是否存在 event 起点
  - `future_bottleneck_object_id` / `type` / `time_to_start` / `duration`
- `root_cause_reason`（无扰动时的规则启发式）：
  - `blocked_downstream` / `starved_upstream` / `high_utilization` / `queue_buildup` / `score_threshold`
  - 有 `disturbance_log` 后应优先对齐扰动真值（阶段 D）

## 5. 验证结果（`2026-07-18_17-46-35_seed42`）

命令已成功执行，摘要：

| 指标 | 值 |
|------|-----|
| `episode_end_s` | 3333 |
| 资源数（含 buffer） | 约 15+ buffer |
| `window_feature_table` 行数 | **5544**（30s+60s） |
| `bottleneck_label` 行数 | **168** |
| `bottleneck_event` 条数 | **1**（阈值较严 + G0 弱瓶颈） |

### 高分窗口常出现的节点（合理）

- `num02_rollerbedCNCPipeIntersectionCuttingMachine_ws0`（切割，产线入口）
- `num04_groovingMachineLarge_ws0`
- `num01_weldingRobot_ws0`
- `num00_rotaryPipeAutomaticWeldingMachine_ws0`

与原始表中 `BLOCKED`/`PROCESSING` 集中工位一致，说明分数方向正确。

### 预测标签偏稀的原因（预期内）

- 默认配置 + rule_based + 5 件订单属于 **G0 结构性弱瓶颈**
- 事件要求「同一节点连续 ≥2 个高分窗口」，因此 event 很少
- `will_bottleneck` 正样本随之偏少（ws=60 约 1.8%）

这不代表管道失败，而是说明：**要做预测训练需要 G1 提高负载或 G2 注入扰动**，以产生更多可分的瓶颈事件。

## 8. 已知局限（第一版）

1. 上下游比例是全局启发式，未建工艺邻接图  
2. `queue_enter` 挂在当前 task 的 `station_id`，与「下游 waiting 工位」不完全同一节点（仍可用于等待时长）  
3. 无扰动时 `root_cause_reason` 仅为规则推断  
4. G0 数据 event / 正样本稀少，不宜直接宣称预测性能
