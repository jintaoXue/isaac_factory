# 环境复杂度提升：G1 负载实验（订单 5 → 15）

> 日期：2026-07-20  
> 实验组：**G1 负载扫描 · P01_order_scale**  
> 对照基线：`2026-07-19_20-59-20_seed42`（订单 5，rule_based，seed 42）

---

## 1. 目标

在**不注入扰动、不改人机数量、不改龙门架数量**的前提下，仅通过**提高订单量**抬高系统负载，观察瓶颈是否从「弱且分散」变为「更强、更可持续」。

本阶段只动第一维：**环境负载（订单量）**。第二维（扰动注入、人机规模等）后续再做。

---

## 2. 只增加订单量，能否构造瓶颈？

**结论：可以，而且这是你当前环境里最自然、成本最低的加压手段之一。**

### 2.1 机制分析

当前 hc_factory 存在多处**结构性容量上限**（与 `03.实验设计_瓶颈维度与扰动.md` 一致）：

| 约束 | 默认值 | 订单增加后的影响 |
|------|--------|------------------|
| 并行在制品上限 `parallel_producing_limit` | 5 | 15 单不能同时开干，**排队 WIP 变长** |
| 龙门架 | 2 台 active | 每道工序前几乎都有物流，**争抢加剧** |
| 工人 | 5 人 | 人工 subtask 串行，**等待与 STARVED 增多** |
| AGV | 2 台 | `have_AGV` 物流链更易 delay |
| Black/Yellow buffer | capacity=6 | 半成品落库压力上升，**blocking / buffer_full** |
| 机器工位 | 多数 1–2 ws | 多 job 争同一热点机（切管机、焊接机器人、工作台） |

订单从 5 → 15 时，系统总工作量约 **×3**，但并行度仍 capped 在 5，因此预期出现：

1. **Queue 类指标上升**：`queue_length_s`、`occupancy_ratio_s`（buffer）、`avg_waiting_time_s`
2. **Process 类指标上升**：热点机器 `active_pct_s`、`current_active_duration_s` 更长
3. **System 类传播增强**：`upstream_blocked_ratio_s`、`downstream_starved_ratio_s`、`route_delay_s`
4. **瓶颈事件更易合并**：连续热窗变多 → `bottleneck_event.csv` 非空概率上升

### 2.2 与基线 run 的对比预期

| 指标 | 基线（订单 5，`2026-07-19`） | 预期（订单 15） |
|------|------------------------------|-----------------|
| Episode 逻辑时长 | ~3434 s | ~8000–12000 s（约 3×，视完成进度） |
| 热窗数 `is_bottleneck_window=1` | 10 / 173 | **明显增加** |
| `bottleneck_event` 条数 | 0（热窗太分散） | **≥1 的可能性大** |
| `will_bottleneck=1` | 0 | 可能仍少，取决于 horizon 与分数分布 |
| buffer 快照 / enqueue 事件 | 1833 snapshot | **占用率更高、周转更频** |
| `resource_event` STARVED/BLOCKED | 234 / 49 | **绝对次数显著上升** |

### 2.3 局限（仅加订单做不到的事）

- **不能**单独制造「设备故障」「缺料 surge」等需扰动模块的根因类型
- **不能**改变 rule_based 的确定性调度偏差（需换策略或 MARL 对照）
- 若 USD 场景里**物理物料实例数不足 15**，会在启动时报 prim 找不到——**已在代码中修复**：`HC_import.usd` 仅含 00–04，加载后 `ensure_product_water_pipe_prims()` 会从模板 duplicate 到 05–14（`src/material.py`）

## 4. 数据采集命令

在仓库根目录、`env_isaaclab` 环境下：

```bash
conda activate env_isaaclab
cd ~/work/isaac_factory

python train.py \
  --task HRTPaHC-v1 \
  --algo rule_based \
  --num_envs 1 \
  --device cuda:0 \
  --headless
```

- 输出目录：`source/isaaclab_tasks/.../output/bottleneck_dataset/<新 run_id>/`
- Episode 会在 `progress.production_done == True`（15 件全部完工）后自动 reset
- 预计单 episode **显著长于** 基线（约 1–3 小时量级，视机器与是否开相机而定）

### 4.1 离线聚合（采完后）

```bash
python3 source/isaaclab_tasks/isaaclab_tasks/direct/hc_factory/tools/build_bottleneck_features.py \
  --run_dir source/isaaclab_tasks/isaaclab_tasks/direct/hc_factory/output/bottleneck_dataset/<新 run_id> \
  --window_sizes 30,60 \
  --horizon 60 \
  --score_threshold 0.55 \
  --min_event_windows 2
```

### 4.2 与基线对比时建议看的字段

- `derived/env_00/pipeline_summary.json`：`hot_windows`、`n_events`、`will_bottleneck_rate`
- `bottleneck_label.csv`：热窗时间分布、主导 `bottleneck_node_t`
- `window_feature_table.csv`：buffer / gantry / machine 的 score 排序
- 原始表：`buffer_event_log.csv` 占用率、`resource_event_log.jsonl` 中 BLOCKED/STARVED 计数

## 5. 验收标准（本实验是否「加压成功」）

满足以下 **任意 2 条** 即可认为订单 15 相对订单 5 有效抬升负载：

1. `pipeline_summary.json` 中 `per_window_size[*].hot_windows` 总数 **> 10**
2. `bottleneck_event.csv` **至少 1 条**合并事件
3. 原始 `job_trace` 行数 **> 500**（约为 5 单时的 2.5× 以上）
4. buffer 平均 `occupancy_ratio`（snapshot 行）**高于** 基线同窗口
5. `resource_event_log` 中 STARVED+BLOCKED 事件总数 **> 基线 283**

若启动失败（RigidPrim / prim path 报错），优先检查 USD 是否支持 15 批物料实例，或临时降到订单 10 做中间点。

**运行时崩溃 `No free storage found for product_00_maded`**：旧逻辑中仓位 `num_material` 只增不减，且入仓每步重复 `+=1`，高订单下 Yellow 仓会被误判为满。已在 `src/material.py` 修复入仓/出仓计数（2026-07-21）。
